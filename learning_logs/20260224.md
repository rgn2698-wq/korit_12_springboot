# RESTful API 명세서 & Swagger(OpenAPI) 정리

## 1. RESTful API 명세서가 필요한 이유
RESTful API는 사용하는 개발자가 기능과 동작을 이해할 수 있도록 **문서화**되어야 한다. 문서에는 보통 아래가 포함된다.

- 사용 가능한 **엔드포인트 목록**
- 허용되는 **데이터 형식(JSON 등)**
- 요청/응답 방식, 인증 방식 등 **API 사용 방법**

특히 우리는 **Spring Data REST**를 적용한 상태라서 엔드포인트들이 자동 생성된다.  
즉, 우리가 명시적으로 만들지 않은 엔드포인트도 생길 수 있으므로 **명세서의 중요성이 더 커진다.**

---

## 2. Swagger / OpenAPI란?
### 2.1 OpenAPI Specification(OAS)
- REST API를 **설계/구축/문서화**하는 방법을 정의한 **표준 사양**

### 2.2 Swagger
- OpenAPI 사양을 기반으로 API 구조를 **시각화**
- API를 **직접 실행/테스트**할 수 있게 지원하는 도구

### 2.3 도입 목적
- 백엔드 ↔ 프론트 협업 시  
  별도 문서 작성 없이 **실시간으로 갱신되는 API 명세서** 제공

---

## 3. API란?
### 3.1 정의
- **Application Programming Interface**
- 서로 다른 소프트웨어가 “대화”할 수 있도록 하는 **규칙/매개체**

비유:
- Client(손님): 앱/웹(요청을 하는 쪽)
- Server(주방): 데이터/기능이 있는 시스템(처리하는 쪽)
- API(점원): 요청을 전달하고 응답을 가져오는 역할

핵심:
- Client는 Server 내부 로직을 몰라도 된다.  
  정해진 규칙(API 규격)에 맞춰 요청하면 결과를 받을 수 있다.

### 3.2 API의 핵심 역할
1. **전달자**: request / response 처리
2. **보안/관문**: 인증된 사용자만 접근 허용 (API Key 등)
3. **표준화**: Java/Python 등 서로 다른 시스템도 동일 규칙으로 소통 가능

> 우리가 만든 백엔드에서는 `Controller` 클래스가 API의 “입구(관문)” 역할이다.

## 4. 프로젝트 설정 (Swagger / OpenAPI 적용)

### 4.1 의존성 추가 (SpringBoot 3.x 주의)
SpringBoot 3.x에서는 `javax`가 아니라 **`jakarta`** 기반이다.

Gradle 의존성:
```gradle
implementation 'org.springdoc:springdoc-openapi-starter-webmvc-ui:2.8.15'
```

- `starter` 포함된 버전 사용 권장 (오류 방지)
- Maven 검색 시에도 `starter-webmvc` 포함 여부로 구분

---

### 4.2 application.properties 설정
Swagger UI 경로, OpenAPI 문서 경로 등을 지정한다.

> 현재는 **Spring Data REST** 때문에 basePath가 `/api`로 잡혀있을 수 있으니 유의.

```properties
# Swagger UI access path (default: /swagger-ui/index.html)
springdoc.swagger-ui.path=/swagger-ui.html

# OpenAPI docs path
springdoc.api-docs.path=/api-docs

# Spring Data REST와 통합
springdoc.show-data-rest=true
```

접속 예:
- 기본: `http://localhost:8080/swagger-ui/index.html`

---

### 4.3 API 문서 상세 설정 (@Bean 사용)
Swagger 문서의 제목/설명/버전을 커스터마이징하려면 설정 클래스를 만든다.

#### `@Bean`의 의미
- 메서드가 반환하는 객체를 **Spring IoC 컨테이너가 관리하는 빈(bean)** 으로 등록
- 주로 `@Configuration` 클래스 내부에서 사용
- 외부 라이브러리 객체 등록/복잡한 초기화 로직 등에 적합

예시:
```java
package com.korit12.cardatabase.config;

import io.swagger.v3.oas.models.OpenAPI;
import io.swagger.v3.oas.models.info.Info;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class OpenApiConfig {

    @Bean
    public OpenAPI cardatabaseOpenAPI() {
        return new OpenAPI()
                .info(new Info()
                        .title("Car Database API")
                        .description("자동차 및 소유주 관리를 위한 REST API 명세서")
                        .version("1.0.0")
                );
    }
}
```

동작 흐름 요약:
1. `@Configuration` 클래스 스캔 → 설정 소스로 인식  
2. `@Bean` 메서드 실행 → 반환 객체를 빈으로 등록  
3. Swagger UI가 이 빈을 읽어 API 문서 정보로 사용

## 5. Bean(빈) 정리 (시험 대비)

### 5.1 빈이란?
- 스프링 IoC 컨테이너가 생성하고 관리하는 자바 객체
- 개발자가 `new`로 만드는 일반 객체와 달리, **생명주기(생성/초기화/소멸)** 가 컨테이너에 의해 관리된다.

### 5.2 핵심 역할: DI(의존성 주입)
- 스프링은 `ApplicationContext`를 통해 빈을 관리하며  
  필요한 곳에 빈을 주입해준다 (필드/생성자/세터)

예:
- `CarRepository carRepository = new CarRepository();`를 직접 호출하지 않아도  
  스프링이 빈을 만들어 주입할 수 있다.

### 5.3 빈 등록 방법
- **컴포넌트 스캔(자동)**:
  - `@Component`, `@Service`, `@Repository`, `@Controller`, `@RestController`
- **수동 등록**:
  - `@Configuration` + `@Bean`

### 5.4 특징
- 기본적으로 **싱글톤**으로 하나만 생성하여 공유
- 빈은 이름을 가지며, 스프링이 해당 이름/타입을 기준으로 주입한다.

## 6. 백엔드 보호(Spring Security)

### 6.1 Spring Security란?
- Java 웹 애플리케이션을 위한 대표적인 보안 라이브러리

#### 기본 동작(디폴트)
1. 인메모리 사용자 1명 자동 생성 (`user`, 비밀번호는 콘솔 출력)
2. 정적 리소스(`/css`, `/images`)는 보안 제외
3. 그 외 모든 엔드포인트는 인증 요구
4. 기본 보안 기능 활성화 (HSTS/XSS/CSRF 등)
5. 기본 로그인 페이지 자동 생성

---

### 6.2 인메모리 로그인 고정(SecurityConfig)
디폴트 비밀번호는 실행할 때마다 바뀐다.  
고정하려면 설정 클래스를 만든다.

```java
package com.korit12.cardatabase.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.security.provisioning.InMemoryUserDetailsManager;

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public InMemoryUserDetailsManager userDetailsService() {
        UserDetails user = User.builder()
                .username("user")
                .password(passwordEncoder().encode("password"))
                .roles("USER")
                .build();

        return new InMemoryUserDetailsManager(user);
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

### 6.3 인메모리 → DB 기반 로그인(AppUser)
인메모리는 서버 재시작 시 초기화된다.  
DB에 저장하려면 `AppUser` 엔티티와 리포지토리를 만든다.

#### AppUserRepository 예시
```java
package com.korit12.cardatabase.domain;

import org.springframework.data.jpa.repository.JpaRepository;
import java.util.Optional;

public interface AppUserRepository extends JpaRepository<AppUser, Long> {
    Optional<AppUser> findByUsername(String username);
}
```

#### Optional을 쓰는 이유
- username은 보통 `unique`라 결과는 **0개 또는 1개**
- `null` 처리로 인한 NPE 방지를 위해 `Optional` 사용

### 6.4 UserDetailsService 구현(UserDetailsServiceImpl)
스프링 시큐리티는 `UserDetails` 타입을 요구한다.  
따라서 DB에서 가져온 `AppUser`를 `UserDetails` 형태로 변환해야 한다.

```java
package com.korit12.cardatabase.service;

import com.korit12.cardatabase.domain.AppUser;
import com.korit12.cardatabase.domain.AppUserRepository;
import org.springframework.security.core.userdetails.*;
import org.springframework.stereotype.Service;

import java.util.Optional;

@Service
public class UserDetailsServiceImpl implements UserDetailsService {

    private final AppUserRepository appUserRepository;

    public UserDetailsServiceImpl(AppUserRepository appUserRepository) {
        this.appUserRepository = appUserRepository;
    }

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        Optional<AppUser> user = appUserRepository.findByUsername(username);

        if (user.isPresent()) {
            AppUser currentUser = user.get();
            return User.withUsername(username)
                    .password(currentUser.getPassword())
                    .roles(currentUser.getRole())
                    .build();
        }

        throw new UsernameNotFoundException("해당 username을 가진 사용자를 찾지 못했습니다.");
    }
}
```

### 6.5 SecurityConfig에서 DB 기반 인증 사용
인메모리 대신 `UserDetailsServiceImpl`을 사용하도록 설정

```java
package com.korit12.cardatabase.config;

import com.korit12.cardatabase.service.UserDetailsServiceImpl;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.authentication.builders.AuthenticationManagerBuilder;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.context.annotation.Bean;

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    private final UserDetailsServiceImpl userDetailsService;

    public SecurityConfig(UserDetailsServiceImpl userDetailsService) {
        this.userDetailsService = userDetailsService;
    }

    public void configureGlobal(AuthenticationManagerBuilder auth) throws Exception {
        auth.userDetailsService(userDetailsService);
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

---

### 6.6 BCrypt 관련 메모
- BCrypt는 해싱 알고리즘  
- 예: `$2a$12$...`
  - `$2a`: 알고리즘 버전
  - `$12`: 강도(cost)
- Spring Security 기본 강도는 보통 10

---

### 6.7 Spring Data REST 자동 생성 엔드포인트 보안 이슈
Spring Data REST를 쓰면 `/api/appUsers` 같은 엔드포인트가 자동 생성될 수 있다.

문제:
- 로그인만 되어 있어도 다른 사용자들의 **해시된 password** 등을 조회 가능할 수 있다.
- 따라서 **AppUser 관련 엔드포인트는 접근 제한**이 필요하다.
